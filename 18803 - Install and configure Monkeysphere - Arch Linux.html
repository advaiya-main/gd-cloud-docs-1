Install and configure Monkeysphere - Arch Linux

Difficulty: 2
Time: 1 hour

In this article, you will learn how to install and configure Monkeysphere which will allow you to have a trusted SSH login.

Prerequisites

You should have 3 droplets for configuring and testing purpose.

  • server.coolexample.com: The SSH server to which you connect.
  • admin.coolexample.com: Administrator computer that is used to configure access.
  • client.coolexample.com: The client from which you initiate the connection.
  • The SSH server should have domain name which is publically accessible.

    Instead of server key, you are using GPG's web of trust model here.

Install dependencies

  1. Install dependency packages.
    sudo pacman -Sy
    sudo pacman -S --needed base-devel
    sudo pacman -S perl-crypt-openssl-rsa perl-digest-sha1 openssh cpio lockfile-progs socat
  2. Install gnupg version 1 from AUR since Monkeysphere needs gnupg version 1.
    mkdir build
    cd build/
    wget https://aur.archlinux.org/cgit/aur.git/snapshot/gnupg1.tar.gz
    tar -xvzf gnupg1.tar.gz
    cd gnupg1
    makepkg -s

    If you get an error like FAILED (unknown public key 2071B08A33BD3F06), then edit ~/.gnupg/gpg.conf to allow downloading keys from key server.

    Edit file ~/.gnupg/gpg.conf. Uncomment line 159.

    keyserver-options auto-key-retrieve

    Save file and then continue building gnupg1.

    makepkg -s
    sudo pacman -U gnupg1-1.4.19-4-x86_64.pkg.tar.xz

Build GPG keys and authentication

Configure GPG Keys on client and admin droplets. This process include below steps:

  1. Generate GPG Keys.

    This should be performed in both client and admin droplets.

    gpg1 --gen-key

    The GPG key generation process will ask for series of questions for the key to create. Choose 1 to create two RSA keys. Choose 2048 bit encryption. Select ‘O’ so that key won’t expire and finally press ‘Y’ to confirm the details provided.

    Provide user as client in client droplet and admin in admin droplet. Give an email address as client@junkdomain.com or admin@junkdomain.com accordingly.

    It will take some time to complete the key generation process. You can view the key by following commad:

    gpg1/ --list-keys
/home/user/.gnupg/pubring.gpg
----------------------------
pub   2048R/1258D8EF 2015-09-06
uid                  client <client@junkdomain.com>
sub   2048R/FF569918 2015-09-06

    Issue above command in admin droplet and you will see the keys in both droplets.

  2. Upload keys to key-server.

    Find the key id from above command. In this case Key id is 1258D8EF. Perform this step on both client and admin droplets.

    gpg1 --send-key 1258D8EF
  3. Signing the Keys.

    Pull down the opposite key on each of droplets/servers and then sign those keys.

    In Client droplet, get the figure print.

    gpg1 --with-colons --fingerprint client
    tru::1:1441518267:0:3:1:5
pub:u:2048:1:9E0613031258D8EF:2015-09-06:::u:client <client@junkdomain.com>::scESC:
fpr::::::::: BF867BD82FAF8559633336829E0613031258D8EF:
sub:u:2048:1:E0150AD6FF569918:2015-09-06::::::e:

    The portion of the output that is highlighted is the fingerprint for client.

    On the admin droplet, download the key, sign and then upload it to key-server.

    gpg1 --recv-keys BF867BD82FAF8559633336829E0613031258D8EF
gpg1 --sign-key BF867BD82FAF8559633336829E0613031258D8EF
gpg1 --send-key BF867BD82FAF8559633336829E0613031258D8EF

    Do the opposite on admin droplet. Take the fingerprint of admin key and then download, sign and upload the key from client droplet.

    Take the fingerprint from admin droplet.

    gpg1 --with-colons --fingerprint admin

    On the Client droplet, download the key, sign and then upload it to key-server.

    gpg1 --recv-keys BF867BD82FAF8559633336829E0613031258D8EF
gpg1 --sign-key BF867BD82FAF8559633336829E0613031258D8EF
gpg1 --send-key BF867BD82FAF8559633336829E0613031258D8EF

    Refresh both your server keys. You will get the proper result when the propagation completes. Wait for few minutes and then execute below command.

    gpg1 --refresh-keys

    Successful propogation of the keys would show an output stating “new signatures: 1”. If this confirmation message do not appear, you need to wait for some more time and issue the refresh command again.

  4. Establish Trust on administrator's Judgment.

    Through this client will know that the SSH server which it is connecting is trusted by administrator.

    Find current trust settings of client.

    gpg1 --check-trustdb

    Update the trust by executing the below command.

    gpg1 --update-trustdb

    Choose option 4 to trust fully. On successful execution, you will get below output.

    gpg: 3 marginal(s) needed, 1 complete(s) needed, PGP trust model
gpg: depth: 0  valid:   1  signed:   1  trust: 0-, 0q, 0n, 0m, 0f, 1u
gpg: depth: 1  valid:   1  signed:   0  trust: 0-, 0q, 0n, 0m, 0f, 0u

    The value 0-, shows that both keys are trusted.

Install Monkeysphere

Perform below step on all droplets (Client, Admin and on SSH Server).

  1. Install Monkeysphere from AUR.
    cd /home/user/build
    wget https://aur.archlinux.org/cgit/aur.git/snapshot/monkeysphere.tar.gz
    tar -xvzf monkeysphere.tar.gz
    cd monkeysphere
    makepkg -s
    sudo pacman -U monkeysphere-0.37-2-any.pkg.tar.xz
  2. Generate GPG key on SSH Server.
    sudo monkeysphere-host import-key /etc/ssh/ssh_host_rsa_key ssh:// junkdomain.com
  3. Upload the key to key-server.
    sudo monkeysphere-host publish-key

Validate server key as server administrator

  1. Sign the SSH server's GPG key as a server administrator, so that client user can verify server identity by trusting administrator.

    Get the GPG fingerprint on SSH server.

    sudo monkeysphere-host show-key
    pub   2048R/3F68800A 2015-09-06
uid                  ssh://junkdomain.com
OpenPGP fingerprint: 58921230505B96E5D50A8968684414D33F68800A
ssh fingerprint: 2048 35:77:0b:f6:9b:1e:7c:69:45:6a:20:54:ab:ef:84:33 (RSA)
  2. On the admin droplet, execute the below command to receive, sign and upload key to key-server: 
    gpg1 --recv-key 58921230505B96E5D50A8968684414D33F68800A
gpg1 --sign-key 58921230505B96E5D50A8968684414D33F68800A
gpg1 --send-key 58921230505B96E5D50A8968684414D33F68800A

Verifying identity/key of SSH server from client

  1. Execute below command to verify the identity:
    ssh -oProxyCommand='monkeysphere ssh-proxycommand %h %p' junkdomain.com

    This command will give below output.

    -------------------- Monkeysphere warning -------------------
Monkeysphere found OpenPGP keys for this hostname, but none had full validity.
An OpenPGP key matching the ssh key offered by the host was found:
pub   2048R/3F68800A 2015-09-06
uid       [ unknown] ssh://junkdomain.com
sig!3        3F68800A 2015-09-06  ssh:// junkdomain.com
RSA key fingerprint is 35:77:0b:f6:9b:1e:7c:69:45:6a:20:54:ab:ef:84:33.
-------------------- ssh continues below --------------------
The authenticity of host domain.com (<no hostip for proxy command>)' can't be established.
ECDSA key fingerprint is d5:46:6c:0f:eb:a9:d1:e1:02:c6:61:d0:37:ea:98:68.
Are you sure you want to continue connecting (yes/no)?
  2. You are prompted for a verification step. At this stage, you are not sure that you are connecting to the right server. So here you can give no and discontinue the process.

    At this point, you can see that server key is found in your key list.

    gpg1 --list-keys
    /home/user/.gnupg/pubring.gpg
----------------------------
pub   2048R/1258D8EF 2015-09-06
uid                  client <client@junkdomain.com>
sub   2048R/FF569918 2015-09-06
pub   2048R/462908AF 2015-09-05
uid                  admin <admin@junkdomain.com>
sub   2048R/62F4D3E5 2015-09-05
pub   2048R/3F68800A 2015-09-06
uid                  ssh://junkdomain.com

    Wait for some time and issue the below command on all droplets to refresh the keys.

    gpg1 --refresh-keys

    In the output, you will see “new signatures: 1”. This ensures that everything is proper.

  3. Check the signature of the server from client, you can find both admin and server keys.
    gpg --list-sigs ssh://junkdomain.com
    pub   2048R/3F68800A 2015-09-06
uid                  ssh://junkdomain.com
sig 3        3F68800A 2015-09-06  ssh://junkdomain.com
sig          462908AF 2015-09-06  admin admin@junkdomain.com
  4. Issue command to connect to server. This time you won’t get any warning message.
    ssh -oProxyCommand='monkeysphere ssh-proxycommand %h %p' junkdomain.com
    user@server.coolexample.com's password:

Make client-server connection easier

  1. Add the following command in client’s ssh configuration file to make the connecting procedure easier.
    sudo vi ~/.ssh/config
    Host *
ProxyCommand monkeysphere ssh-proxycommand %h %p

    This will perform all verification process and you will only get a password prompt.

  2. To test further, you can even clear the ssh known_hosts file and try to ssh, you won’t be asked to accept the hosts.
    ssh server.coolexample.com
    user@server.coolexample.com's password:

Conclusion

In this article you have learnt to install and configure Monkeysphere in Arch Linux to trust the ssh server. With this, if administrator trusts the server, then clients won’t be asked to validate server for connection.
