Deploy Django /PostgreSQL/Gunicorn/NGINX Stack on CentOS 6

Difficulty: 3
Time: 1.50

It is required that you have 'Python 2.7 installed on your system before following this guide. By default 'CentOS 6 runs' Python 2.6.6 as the system Python, which will not work. If you have not already done so, you will want to review Install and Configure Different Versions of Python - 'CentOS 6.

Solution Stacks

This guide will cover how to deploy a software solution stack for deploying 'Django applications on your Self-Managed 'CentOS 6 server, we have decided to focus our guide around one of the more stable and most-popular solution stacks for 'Django:

  • Web server - NGINX
  • WSGI Application server - Gunicorn
  • Database - PostgreSQL
  • Python - Virtual Environment (virtualenv)

Lastly, this guide will be focused on deploying 'Django using 'Python 2.7. This is simply for the sake of ease, as our new ''Linux servers come deployed with 'Python 2.7 out of the box, and 'Django is compatible with 2.7. You may if you wish set up a separate installation of Python 3. The steps should be mostly the same, though the names to the interpreters and paths may change.

Installation and Setup

We first need to update all of the RPM & YUM packages

  1. Log into your server through SSH
    ssh username@ip_address_DNS
  2. Access the root user by running
    sudo su -
  3. Run the following command to update your yum cache:
    yum -y update
  4. Run the following command to update your yum cache:
    yum -y update

Install Postgresql

  1. As root, run the following command to install PostgreSQL and the appropriate packages:
    yum -y install postgresql postgresql-contrib postgresql-plpython postgresql-devel
  2. This will install PostgreSQL and a few appropriate modules. Then, we just need to to initialize the database:
    service postgresql initdb
  3. And we are now okay to start the postgresql service.
    service postgresql start
  4. Before we proceed further, we need to make some changes to the PostgreSQL configuration. This is to make sure we can allow password authentication for database users. This will save us a step later.

    Open the PostgreSQL configuration file in the text editor of your choice. For our example, we will use vim:
    vim /var/lib/pgsql/data/pg_hba.conf
  5. Scroll down and you will see lines for defining the local, IPv4, and IPv6 connection sockets. By default, you should see this: 
    # TYPE  DATABASE        USER            ADDRESS                 METHOD
# "local" is for Unix domain socket connections only
local   all             all                                     peer
# IPv4 local connections:
host    all             all             127.0.0.1/32            ident
# IPv6 local connections:
host    all             all             ::1/128                 ident
  6. Using your text editor, change the peer and ident entries to trust, then save and close the file. Yours should look like the example below: 
    # TYPE  DATABASE        USER            ADDRESS                 METHOD
# "local" is for Unix domain socket connections only
local   all             all                                     trust
# IPv4 local connections:
host    all             all             127.0.0.1/32            trust
# IPv6 local connections:
host    all             all             ::1/128                 trust
  7. Restart PostgreSQL so that these changes take:
    service postgresql restart

  2. Now that we have that done, lets go ahead and use yum to install nginx:
    yum -y install nginx

Install Setuptools

Note: Last warning - do not proceed with the rest of the guide until you have installed at LEAST Python 2.7. We cannot use the system Python (2.6.6) for anything further in the guide.

  • First lets confirm 100% that we are running Python 2.7 or greater. This guide assumes you have followed the instructions here and installed Python 2.7. So any calls to python will instead be to python2.7.
    [root@server Python-2.7.6]# python2.7 -V
    Python 2.7.6
    Now that we have confirmed that, we are safe to install the python-devel, virtualenv, and setuptools modules. We cannot use yum to do this since we compiled Python 2.7.6 from source. So we'll manually do it. Lets start with setuptools, which will make the rest of this easy.
  • Run the following command to grab the source for the version of setuptools we want for Python 2.7, and install it.
    curl https://bitbucket.org/pypa/setuptools/raw/bootstrap/ez_setup.py | python2.7 -

Pip - Install

Pip is a package manager for Python application packages. It will help us to install Gunicorn as well as Django.

  • To install pip, run the following command:
    curl https://raw.githubusercontent.com/pypa/pip/master/contrib/get-pip.py | python2.7 -

VirtualEnv - Install

  • Next, we need to install the virtualenv module into Python. Since we installed pip, this should make it pretty easy.
    pip install virtualenv

VirtualEnv - Create Environment

Now that virtualenv is installed, we are ready to create our new virtual environment. We will set this up inside the default /var/www directory in order to maintain an easy naming convention but in reality, you can use whichever path you prefer. Simply make sure that you are replacing our example paths with any paths you will be using.

  1. Create your directory for the application:
    sudo mkdir /var/www/django_dev
  2. Lets make your new user the owner of this directory:
    sudo chown django_dev:django_dev /var/www/django_dev
  3. Move to the directory for your app
    cd /var/www/django_dev
  4. Now we are ready to create our virtual environment. You may choose any name you wish for your virtual environment, for the sake of simplicity we are naming ours
    django_dev_venv:
    virtualenv django_dev_venv
    You should see a success message that the virtual environment has been created, and installed pip and setuptools
  5. Let’s also make a directory for static files. This will save us a step later:
    sudo mkdir /var/www/django_dev/django_dev_venv/static/
  6. Lastly, we need to activate the interpreter inside the virtual environment so that it may be used:
    source django_dev_venv/bin/activate
  7. You should now be running commands inside your virtual environment. You will know this when the name of your virtualenv is listed in parenthesis before your normal command prompt, like so:
    (django_dev_venv)[django_dev@server django_dev]$
    Now that we have the virtual environment initialized, and a place to put our application we can move on to our app server.

Configure PostgreSQL

Lets go ahead and start by configuring our PostgreSQL database server, creating a database and getting it ready to tie into Django. Make sure you are keeping track of any database usernames, database names, and passwords during this process so that you can use these for connection strings later

  1. First, lets change to our PostgreSQL user:
    sudo su - postgres
    This will change you to the postgres user.
  2. Run this command to create your database. You can give your database any name you wish; for our example we will be naming our django_user:
    createdb django_user
  3. Create your database user. You can use which ever user you wish here, just make sure to keep track of what you use for later:
    createuser dbuser -P
    You will be met with prompts for the username and the password. It will also prompt a few other more advanced options - say no to all of them for right now. We are setting up this user with the most basic of permissions.
  4. Next we want to initiate the PostgreSQL command-line interface:
    psql
  5. You should see a prompt that looks like this:
    postgres=#
  6. Now that we are here, the last thing we need to do is grant full privileges for your user to your database. Using our example:
    GRANT ALL PRIVILEGES ON DATABASE django_user TO dbuser;
    You now have a PostgreSQL database and a user to access that database with. Now we can configure Django to use this database.
  7. Exit out of the PostgreSQL command by typing
    \q
  8. Then exit back to your virtual environment by typing
    exit

Configure Gunicorn

Please take the time to review Gunicorn's documentation page for configuring Gunicorn to meet your application's needs.

  1. Run the following command to start the Gunicorn application server over port 8000.
  2. gunicorn myproject.wsgi:application --bind domainoripaddress.tld:8000
  3. You can replace 'domainoripaddress' with either a domain name with public DNS pointed at the server, or simply the IP address of your server - whichever works for you right now. Also replace 'myproject' with the name of the Django project we created earlier. So to demonstrate the example we are using:
    gunicorn django_dev.wsgi:application --bind django_dev.com:8000
  4. Create a configuration file directly in your virtual environment directory:
    vim /var/www/django_dev/django_dev_venv/gunicorn_config.py
  5. To create a configuration file that exactly matches our example, this is what we would put into the file:
    command = '/var/www/django_dev/django_dev_venv/bin/gunicorn'
    pythonpath = '/var/www/django_dev/django_dev_venv/django_dev'
    bind = 'django_dev.com:8000'
    workers = 3
    user = 'django_dev'
  6. Replace the domain and path with the domain and path you wish to use. Make sure that pythonpath is equal to the Django project directory, and that command is equal to the Gunicorn executable in your virtual environment. Once done, save and close the file.
  7. We are going to start our server again, this time with a different command:
    sudo /var/www/django_dev/django_dev_venv/bin/gunicorn -c /var/www/django_dev/django_dev_venv/gunicorn_config.py django_dev.wsgi

    Note: This specifies the path to gunicorn, the path to the config file, and the name of the Django project WSGI. The -c flag indicates that we are using a configuration file.

    "
  8. Re-open the gunicorn_config.py file in your text editor and change your bind to be
    bind = '127.0.0.1:8000'
  9. Once done, save and close the file.

Configure NGINX

We're almost done with our stack. Last thing we need to do here is set up NGINX to act as a reverse-proxy server. Gunicorn will serve the Django application to NGINX, which will be then sent to the rest of the internet.

  1. Before we begin, lets take care of a few administration tasks. We're done for now in the virtual environment, so get back to root by typing
    exit
  2. Once you're logged in as root, we need to shut Apache off to make room for our NGINX server:
    service httpd stop
  3. Since we’ll be using NGINX from here on out, we want to make sure to tell the server not to start Apache upon server reboot. Later, we’ll tell it to use NGINX instead.
    chkconfig httpd off
  4. Then we can fire up NGINX, if you haven't already:
    service nginx start
  5. Configure NGINX to start upon boot:
    systemctl enable nginx
  6. Once done, confirm that it is running by typing
    service nginx status
    You should see an output related to the NGINX server being active and running. We are now ready to start configuring NGINX.
  7. We need to configure NGINX to serve those static files. Create and open up a new NGINX config file with the following commands. Our neq config file will simply use the name of the project. We are using vim, but you can use whichever text editor you prefer:
    mkdir -p /etc/nginx/sites-available vim /etc/nginx/sites-available/django_dev.con
  8. We will create a new NGINX server block. Our example will use the domain and folder paths we laid out in our working example: 
    server {
        server_name django_dev.com www.django_dev.com;

        access_log off;

        location /static/ {
            alias /var/www/django_dev/django_dev_venv/static/;
        }

        location / {
                proxy_pass http://127.0.0.1:8000;
                proxy_set_header X-Forwarded-Host $server_name;
                proxy_set_header X-Real-IP $remote_addr;
                add_header P3P 'CP="ALL DSP COR PSAa PSDa OUR NOR ONL UNI COM NAV"';
        }
    }
  9. Lets set up our symbolic link so that NGINX knows that this site is enabled. Do so by running the following commands:
    mkdir -p /etc/nginx/sites-enabled
    ln -s /etc/nginx/sites-available/django_dev.conf /etc/nginx/sites-enabled/django_dev.con
  10. Open the NGINX configuration file in a text editor:
    vim /etc/nginx/nginx.conf
  11. Add these lines to the end of the http {} block, then save the file:
    include /etc/nginx/sites-enabled/*.conf;
    server_names_hash_bucket_size 64;
  12. Finish it off by restarting NGINX so that it takes the new changes.
    service nginx restart
    We now have NGINX setup as our reverse-proxy.

Putting this all together

  1. Let’s get our Gunicorn server running in the background. Lets first move back to our django_dev user, activate our virtual environment, and start a screen. You can do this by running the example commands in the following order:
    su django_dev -
    source /var/www/django_dev/django_dev_venv/bin/activate
    sudo screen
    You should now have a new screen open.
  2. Navigate to the application directory:
    cd /var/www/django_dev/django_dev
  3. Start your Gunicorn server:
    sudo /var/www/django_dev/django_dev_venv/bin/gunicorn -c /var/www/django_dev/django_dev_venv/gunicorn_config.py django_dev.wsgi
    Your Gunicorn server should now be running with three worker processes under 127.0.0.1:8000
  4. Close the screen and go back your main prompt by hitting CTRL-A, then D, on the keyboard.
  5. Restart NGINX one last time.
    sudo service nginx restart

Conclusion

There may be additional steps you need to take to configure your specific application to work with NGINX and PostgreSQL. The ultimate authority for all configuration information regarding these elements will be the official documentation pages for PostgreSQL, Gunicorn, NGINX, and Django.
