Optimize Unicorn for your Ruby on Rails application - Debian, Ubuntu

Difficulty: 3
Time: 1.5

Rails is a nice web framework, but over past few years, it has grown into a complex system. Particularly when using Unicorn, Rails apps tend to consume a lot of memory. Understanding its operational details and optimizing its performance can be a critical factor for success of any Rails developer. This article will assist you with few simple things that can be done to enhance the performance of Unicorn for your Rails applications on Debian or Ubuntu platform.

Prerequisite

  1. Install Ruby on Rails using RVM - Debian.
  2. Install Ruby on Rails using RVM - Ubuntu.

This article assumes that user is logged in with sudo user privileges.

Set Unicorn as a default option

By default, Rails uses the Webrick server to run applications. It is fast, but unfortunately, it can not handle concurrent connection requests. For optimized performance, Unicorn must be set as the default server for Rails. Unicorn is a multi-threaded web server, capable of handling multiple requests simultaneously, thus optimizing the performance considerably. For details about setting up Rails app with Unicorn and NGINX, see:
  • Deploy a Rails App with Unicorn and NGINX - Debian.
  • Deploy a Rails App with Unicorn and NGINX - Ubuntu.

Upgrade to 2.x

If you are using ruby 1.9 or earlier, it is recommended to upgrade to 2.x. Because ruby 1.x and earlier versions do not support automated garbage collection, and thus are unable to take benefits of Copy-on-Write (CoW) feature.

  1. Upgrade Ruby from 1.9 to 2.x using the following command:
    rvm install 2.2.3
  2. Verify the Ruby version using the following command:
    ruby -v
    #ruby 2.2.3p173 (2015-08-18 revision 51636) [x86_64-linux]

Fine-tune the configuration settings

To boost the performance of Unicorn, some configuration settings can be checked. These are available in the Unicorn's basic configuration file config/unicorn.rb.

  1. Open the unicorn.rb in your favourite editor using the following command:
    vim config/unicorn.rb
  2. Update the desired configuration variables based on requirements. 
    app_path = File.expand_path(File.dirname(__FILE__) + '/..')
worker_processes 2
preload_app true
timeout 30
  3. Save and exit the file by pressing the Esc key and then typing :wq, followed by the Enter key.

Below is the explanation of the setting variables:

  • app_path

    Set the current app's path for later reference. Rails.root isn't available at this point, so point up a directory.

  • worker_process

    Worker process defines the total number of concurrent worker to be launched. This ideally depends upon the memory available to the processes to execute. So depending upon your application and the availability of memory, you can set the number of concurrent worker process to run. For example, to allow 5 concurrent connections, you can set the value to 5 workers.

  • preload_app

    This defines the ability to preload the Rails application using CoW, before forking the worker processes. This setting impacts startup time when running Unicorn worker processes. This should be set to true. Although, when using this option, special care is to be taken when opening and closing any sockets. When workers are forked, any associated connection (like database connections or others which require sockets) must be closed and reopened.

  • timeout

    Timeout defines the time duration to wait before the worker times out. It is usually preferred to keep a lower base value for the timeout to avoid holding back a long queue. So ideally 15 to 30 is considered as a rational value, but depending upon the nature of applications, this can be changed accordingly.

Deploy Unicorn Worker Killer

When using Unicorn, all the forked or child processes of Ruby application consumes memory. So in case there is any memory leakage in the Ruby application, use of Unicorn would increase the problem multi-fold times. Over a period of time,situation may arrive when all the memory might get used and no additional memory is left, ultimately causing the system to halt or crash. Although Unicorn may not be causing this problem, but can be resolved using Unicorn-worker-killer.

Setup Unicorn Worker Killer

Unicorn worker killer is a gem that helps perform self-check of memory consumed by unicorn workers. In case the memory consumption of the unicornm workers exceeds due to any memory leakage, it provides automatic restart of worker without affecting any request. It is based on the maximum number of requests and the consumed memories, and restarts the worker in case the consumption exceeds the pre-defined limits. Here are the steps for using unicorn-worker-killer:

  1. Install Unicorn Worker Killer.
    gem install 'unicorn-worker-killer'
    bundle install
  2. Open file config.rb located in the default project directory.
    sudo vim config.rb
  3. Update the variables max_requests_min and max_requests_max to the desired values. 
    	
  # --- Start of unicorn worker killer code ---

  require 'unicorn/worker_killer'

  max_request_min = 500
  max_request_max = 600

  # Max requests per worker
  use Unicorn::WorkerKiller::MaxRequests, max_request_min, max_request_max

  oom_min = (240) * (1024**2)
  oom_max = (260) * (1024**2)

  # Max memory size (RSS) per worker
  use Unicorn::WorkerKiller::Oom, oom_min, oom_max, ,versbose=true

  # --- End of unicorn worker killer code ---
	
   require ::File.expand_path('../config/environment',  __FILE__)
   run YourApp::Application
    The variables max_requests_min and max_requests_max defines the minimum and the maximum number of requests per worker. A random number between max_requests_min and max_requests_max is defined as the actual limit. This randomness prevent all workers getting killed simultaneously. The workers are automatically killed and restarted when the number of requests handled by the worker exceeds the limit.
  4. Save and exit the file by pressing the Esc key and then typing :wq, followed by the Enter key.

Conclusion

Unicorn is a painless way to achieve concurrency and performance in your Rails apps whether used with thread safe or not. It provide essential stability to Rails Apps. Unicorn worker killer gracefully restart your application and resolve memory leaks.
