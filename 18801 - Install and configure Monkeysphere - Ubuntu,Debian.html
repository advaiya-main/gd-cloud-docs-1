Install and configure Monkeysphere - Debian, Ubuntu

Difficulty: 2

Time: 6 hours

Monkeysphere is a utility suite that provides a framework for TSL/SSL communication authentication using simple tools like web browser and secure shell. The Monkeysphere system uses GPG keys to validate the servers to users. In this article, you will install and configure Monkeysphere which will allow you to have a trusted SSH login.

Prerequisites

You should have 3 droplets for configuring and testing purpose.

  • server.coolexample.com: The SSH server to which you connect.
  • admin.coolexample.com: Administrator computer that is used to configure access.
  • client.coolexample.com: The client from which you initiate the connection.
  • The SSH server should have domain name which is publically accessible.

Set up GPG keys

The GPG keys must be setup and configured on the client and admin droplets. For this, follow the steps below on both the client and the admin droplet.

  1. Initiate the GPG key generation process.
    gpg --gen-key

    This will begin the process of GPG key generation where a series of questions will be asked, based on which key pair is created.

  2. When asked for the type of keys, choose option 1 to create two RSA keys.
  3. When asked about the length of key, choose the default option to select 2048-bit encryption
  4. When asked about key expiry, choose O to select option key never auto expire.
  5. Press 'Y' to confirm the details provided so far.
  6. When asked about the information on user, provide the following information:
    • For the client droplet, provide username as client and email address as client@junkdomain.com
    • For the admin droplet, provide username as admin and email address as admin@junkdomain.com

    Based on the provided information, a random key will be generated, which might take some time.

  7. View the key using the following command one by one on both droplets:
    gpg --list-keys
  8. Successful generation of key would result into the following output: 
    /home/user/.gnupg/pubring.gpg
----------------------------
pub   2048R/1258D8EF 2015-09-06
uid                  client <client@junkdomain.com>
sub   2048R/FF569918 2015-09-06

GPG key authentication

  1. Identify and note down the key id for both the droplets. The key id can be identified from the output of the previous command gpg --list-keys. In this case, the key was 1258D8EF. 
    /home/user/.gnupg/pubring.gpg
----------------------------
pub   2048R/1258D8EF 2015-09-06
uid                  client <client@junkdomain.com>
sub   2048R/FF569918 2015-09-06
  2. Upload both the keys to key-server, by performing the following steps on both client and admin droplets:
    gpg --send-key 1258D8EF

    Uploading keys usually require few minutes, so you might have to wait for some time.

Sign the client key with admin key

  1. On the client droplet, get the finger print of the GPG key for the client key, by using the following command:
    gpg --with-colons --fingerprint client
  2. Successful execution would result into following output. In this, the long alpha-numeric string just after fpr is the fingerprint of the key. In this case, it is BF867BD82FAF8559633336829E0613031258D8EF 
    tru::1:1441518267:0:3:1:5
pub:u:2048:1:9E0613031258D8EF:2015-09-06:::u:client <client@junkdomain.com>::scESC:
fpr:::::::::BF867BD82FAF8559633336829E0613031258D8EF:
sub:u:2048:1:E0150AD6FF569918:2015-09-06::::::e:

    The portion of the output that is highlighted is the fingerprint for client.

  3. On the admin droplet, download the key and sign the key.
    gpg --recv-keys BF867BD82FAF8559633336829E0613031258D8EF
    gpg --sign-key BF867BD82FAF8559633336829E0613031258D8EF
  4. Upload the signed key back to the key-server.
    gpg --send-key BF867BD82FAF8559633336829E0613031258D8EF

Sign the admin key with client key

  1. On the admin droplet, get the finger print of the GPG key for the admin key, by using the following command:
    gpg --with-colons --fingerprint admin
  2. On Client droplet, download the key, sign and then upload it to key-server.
    gpg --recv-keys BF867BD82FAF8559633336829E0613031258D8EF
    gpg --sign-key BF867BD82FAF8559633336829E0613031258D8EF
    gpg --send-key BF867BD82FAF8559633336829E0613031258D8EF

    Propagation of these changes takes some time, so you might need to wait for few minutes.

  3. Refresh both the server keys using below command to ensure that the new key is accepted on that machine:
    gpg --refresh-keys
  4. Successful propagation of the keys would show a confirmation message as below upon refresh:
    new signatures: 1

    If this confirmation message do not appear, you need to wait for some more time and issue the refresh command again until confirmation is received.

Establish trust on admin

To ensure that the client droplets fully trust admin droplets, you need to establish the trust of the client droplet on the admin droplet.

  1. Check the existing trust setting of the client using the below command on client droplet.
    gpg --check-trustdb
  2. Update the trust setting of client droplet.
    gpg --update-trustdb
  3. Choose option 4 to trust fully.
  4. Successful execution would show the following output: 
    gpg: 3 marginal(s) needed, 1 complete(s) needed, PGP trust model
gpg: depth: 0  valid:   1  signed:   1  trust: 0-, 0q, 0n, 0m, 0f, 1u
gpg: depth: 1  valid:   1  signed:   0  trust: 0-, 0q, 0n, 0m, 0f, 0u

    In the trust section, the value 0 - shows that the keys are trusted.

Install Monkeysphere and configure SSH server

  1. Install Monkeysphere on all the three droplets (Client, Admin and on SSH Server).
    sudo apt-get update
    sudo apt-get install monkeysphere
  2. On SSH Server, generate the GPG key.
    sudo monkeysphere-host import-key /etc/ssh/ssh_host_rsa_key ssh://server.coolexample.com
  3. Upload the key to key-server:
    sudo monkeysphere-host publish-key

Validate the server key as the server administrator

The GPG key of the SSH server should be signed as a server administrator, so that client user can verify server identity by trusting administrator.

  1. Get the GPG fingerprint on SSH server:
    sudo monkeysphere-host show-key
  2. Note the OpenPGP fingerprint provided by the SSH server, which is same as the GPG fingerprint. In this case, the OpenPGP key is 58921230505B96E5D50A8968684414D33F68800A. 
    pub 2048R/3F68800A 2015-09-06
uid ssh://server.coolexample.com
OpenPGP fingerprint: 58921230505B96E5D50A8968684414D33F68800A
ssh fingerprint: 2048 35:77:0b:f6:9b:1e:7c:69:45:6a:20:54:ab:ef:84:33 (RSA)
  3. On the admin droplet, execute the below command to receive the key:
    gpg --recv-key 58921230505B96E5D50A8968684414D33F68800A
  4. Sign the key on the admin droplet.
    gpg --sign-key 58921230505B96E5D50A8968684414D33F68800A
  5. Upload the key back to the key-server.
    gpg --send-key 58921230505B96E5D50A8968684414D33F68800A

Verifying the identity of the SSH server from the client

  1. On the client droplet, verify the identity of the SSH server.
    ssh -oProxyCommand='monkeysphere ssh-proxycommand %h %p' server.coolexample.com
  2. Type no when prompted with the Monkeysphere warning. This warning states that although it has found an OpenPGP key, but full validity is not available. Since it is not confirm that you are connecting to the right server, the connection request must be denied at this point. 
    -------------------- Monkeysphere warning -------------------
Monkeysphere found OpenPGP keys for this hostname, but none had full validity.
An OpenPGP key matching the ssh key offered by the host was found:
pub   2048R/3F68800A 2015-09-06
uid       [ unknown] ssh://server.coolexample.com
sig!3        3F68800A 2015-09-06  ssh://server.coolexample.com
RSA key fingerprint is 35:77:0b:f6:9b:1e:7c:69:45:6a:20:54:ab:ef:84:33.
-------------------- ssh continues below --------------------
The authenticity of host 'server.coolexample.com (<no hostip for proxy command>)' can't be established.
ECDSA key fingerprint is d5:46:6c:0f:eb:a9:d1:e1:02:c6:61:d0:37:ea:98:68.
Are you sure you want to continue connecting (yes/no)?
  3. Check the list of available keys on the client machine.
    gpg --list-keys
  4. Note that the SSH Server key is in the list.
    /home/nik/.gnupg/pubring.gpg
    ----------------------------
pub   2048R/1258D8EF 2015-09-06
uid                  client <client@junkdomain.com>
sub   2048R/FF569918 2015-09-06
pub   2048R/462908AF 2015-09-05
uid                  admin <admin@junkdomain.com>
sub   2048R/62F4D3E5 2015-09-05
pub   2048R/3F68800A 2015-09-06
uid           ssh://server.coolexample.com
  5. Wait for few minutes, and then refresh keys on the client droplet.
    gpg --refresh-keys
  6. Successful propagation of the keys would show a confirmation message as below upon refresh:
    new signatures: 1
  7. Check the signature available on the key of SSH server:
    gpg --list-sigs ssh://server.coolexample.com
  8. Successful propagation of the keys would show an output similar to following: 
    pub   2048R/3F68800A 2015-09-06
uid                  ssh://server.coolexample.com
sig 3        3F68800A 2015-09-06  sshserver.coolexample.com
sig          462908AF 2015-09-06  admin admin@junkdomain.com
  9. Connect to server again. This this time you should not get any warning message.
    ssh -oProxyCommand='monkeysphere ssh-proxycommand %h %p' server.coolexample.com
    user@server.coolexample.com's password:

Make Client-Server connection easier

  1. On the client droplet, open the ssh config file.
    sudo vim ~/.ssh/config
  2. Add the following code anywhere in the file to make the connecting procedure easier. 
    Host *
ProxyCommand monkeysphere ssh-proxycommand %h %p

  3.  Save and exit the file by pressing the Esc key, and then typing :wq, followed by the Enter key.

    This will perform all the verification process and you will only get a password prompt where you need to enter the password.

  4. Test the setup by cleaning up the known_host file.
    rm ~/.ssh/known_hosts
  5. Run the ssh command to connect to the server. This time there should be no warning message from server.
    ssh server.coolexample.com
    user@server.coolexample.com's password:

Conclusion

This is all about installing and configuring Monkeysphere in Debian or Ubuntu droplet to trust the ssh server. With this, if administrator trusts the server, then clients won't be asked to validate server for connection. Now after this, you can use Monkeysphere to authenticate SSH users on Debian or Ubuntu server.
